# 📚 사용자 맞춤형 논문 추천 서비스 — 프로젝트 기획서

> CS/AI 연구자를 위한, 관심사 기반 논문 추천 + LLM 요약 웹 서비스

---

## 1. 서비스 주제

**"내 관심사에 맞는 최신 논문을, 읽기 전에 한눈에 파악할 수 있게."**

CS/AI 분야 대학원생·연구자는 매일 쏟아지는 arXiv 논문 속에서 *무엇을 읽을지* 고르는 데 시간을 많이 씁니다.
본 서비스는 사용자의 **관심 분야**를 기반으로 논문을 추천하고, 각 논문을 **LLM으로 요약**해 주며,
특정 논문과 **비슷한 논문**까지 연결해 주어 "탐색 → 판단 → 저장"의 흐름을 빠르게 만듭니다.

- **타겟 사용자**: CS/AI 대학원생 및 연구자
- **핵심 데이터 소스**: [arXiv](https://arxiv.org) (CS/AI 분야 커버리지가 가장 강함)
- **차별점**: 단순 검색이 아니라 **개인화 추천 + AI 요약 + 유사 논문 연결**의 결합

---

## 2. 핵심 기능 정의

### 2.1 MVP (1주차 목표 범위)

| # | 기능 | 설명 |
|---|------|------|
| F1 | **온보딩 (관심사 선택)** | 사용자가 CS/AI 세부 분야·키워드를 선택. localStorage에 저장 |
| F2 | **추천 피드 (홈)** | 선택한 관심사 기반으로 arXiv 최신 논문 목록 제공 |
| F3 | **논문 상세** | 제목·저자·초록·링크 + **LLM 요약** + **비슷한 논문** 섹션 |
| F4 | **LLM 요약** | Gemini API로 초록을 3~4줄 핵심 요약 (BFF 경유) |
| F5 | **비슷한 논문 추천** | 특정 논문 기준 유사 논문 (Semantic Scholar, BFF 경유) |
| F6 | **검색** | 키워드로 arXiv 논문 검색 |
| F7 | **내 서재 (북마크)** | 관심 논문 저장/조회 (localStorage) |

### 2.2 개인화(추천)의 근거

추천은 두 축으로 구성합니다.

1. **관심 키워드 기반** — 온보딩에서 고른 분야/키워드로 피드를 구성 (F2)
2. **비슷한 논문 기반** — 보고 있는 논문을 기준으로 유사 논문 연결 (F5)

> 행동/로그인 기반 개인화, 추천 점수 학습 등은 **추후 확장(Backlog)** 으로 분리.

### 2.3 추후 확장 (Backlog)

- 북마크·열람 행동 기반 추천 정교화
- 로그인 + 서버 DB로 기기 간 동기화
- 논문 PDF 전문 요약 (초록 → 전문)
- 추천 이유 설명("이 논문을 추천한 이유")

## 3. 기술 스택 및 선택 이유

### 3.1 프론트엔드

| 영역 | 선택 | 선택 이유 |
|------|------|-----------|
| 빌드/프레임워크 | **Vite + React + TypeScript** | 빠른 개발 서버·HMR, 타입 안정성, React 생태계. Agent 기반 코드 생성에도 패턴이 풍부 |
| 라우팅 | **React Router** | 온보딩/홈/상세/검색/서재 등 다중 페이지 구성 |
| 서버 상태 | **TanStack Query** | API 캐싱·로딩·에러·리트라이 자동화 → arXiv/BFF 호출 관리의 핵심 |
| 클라이언트 상태 | **Zustand** | 관심사·북마크 등 가벼운 전역 상태를 단순하게 관리 |
| 스타일링 | **Tailwind CSS** | 빠른 UI 구축, 디자인 일관성, Agent 친화적 클래스 기반 |

### 3.2 백엔드 (얇은 BFF)

| 영역 | 선택 | 선택 이유 |
|------|------|-----------|
| BFF | **Serverless Function** (Vercel Functions / Hono) | 함수 몇 개로 외부 API 프록시. CORS·레이트리밋·**API 키 은닉** 처리 |
| 배포 | **Vercel** | 프론트 + BFF를 한 번에 배포, 환경변수 관리 편리 |

> **BFF를 두는 이유 (중요)**
> - arXiv API는 CORS 헤더가 불안정하고 응답이 **Atom XML** → 프론트에서 직접 쓰기 번거로움
> - **Gemini / Semantic Scholar API 키를 브라우저에 노출하면 안 됨** → 서버에서만 호출
> - 외부 API 레이트리밋·에러를 한 곳에서 흡수

### 3.3 외부 API

| API | 용도 | 호출 위치 |
|-----|------|-----------|
| **arXiv API** | 논문 검색·피드·메타데이터 (메인) | BFF 경유 (XML→JSON 변환) |
| **Semantic Scholar API** | 비슷한 논문 추천 (arXiv ID 매핑) | BFF 경유 |
| **Google Gemini API** | 논문 초록 LLM 요약 | BFF 경유, 키는 서버 `.env` |

> ⚠️ **API 키 보안**: Gemini 키는 `VITE_` 접두사 **없이** 서버 측 환경변수로만 둔다.
> (`VITE_`를 붙이면 클라이언트 번들에 키가 그대로 노출됨)

## 4. 시스템 아키텍처 (개요)

```
[ 브라우저 / React SPA ]
        │  (관심사·북마크는 localStorage)
        ▼
[ BFF (Serverless) ]
   ├── /api/papers     ── arXiv API (검색·피드, XML→JSON)
   ├── /api/similar    ── Semantic Scholar API (유사 논문)
   └── /api/summarize  ── Gemini API (초록 요약, 서버 .env 키)
```

- **개인화 신호(관심사·북마크)** 는 1주차엔 서버 없이 **localStorage** 로 관리
- 외부 API 3종은 모두 **BFF 뒤에 숨김**

## 5. 화면 흐름 / 페이지 구성 초안

### 5.1 화면 흐름

```
 [온보딩]  관심 분야/키워드 선택
     │
     ▼
 [홈 / 추천 피드]  관심사 기반 arXiv 논문 카드 목록
     │   ├──→ [검색]   키워드로 논문 검색
     │   └──→ [내 서재]  북마크한 논문
     ▼
 [논문 상세]
     ├── 제목 · 저자 · 초록 · 원문 링크
     ├── 🤖 LLM 요약 (Gemini)
     ├── 🔗 비슷한 논문 (Semantic Scholar)
     └── ⭐ 북마크 토글
```

### 5.2 페이지 목록

| 페이지 | 경로 | 주요 구성 요소 |
|--------|------|----------------|
| 온보딩 | `/onboarding` | 분야 칩 선택, 키워드 입력, 시작 버튼 |
| 홈(추천 피드) | `/` | 추천 논문 카드 리스트, 관심사 필터, 헤더/네비 |
| 검색 | `/search` | 검색바, 결과 리스트, 정렬 |
| 논문 상세 | `/paper/:id` | 메타정보, LLM 요약, 비슷한 논문, 북마크 |
| 내 서재 | `/library` | 북마크 논문 리스트, 삭제 |

### 5.3 핵심 컴포넌트 (재사용 단위)

- `PaperCard` — 논문 카드 (제목·저자·태그·북마크 버튼)
- `PaperList` — 카드 리스트 + 로딩/빈 상태
- `SummaryBox` — LLM 요약 표시(로딩/에러/재시도)
- `SimilarPapers` — 유사 논문 미니 리스트
- `InterestSelector` — 관심 분야/키워드 선택 UI
- `BookmarkButton` — 북마크 토글
- `SearchBar` — 검색 입력
